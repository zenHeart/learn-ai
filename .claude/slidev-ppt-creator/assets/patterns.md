# 叙事模式 (Narrative Patterns / DSL)

分享必须有一个一句话就能概括的**核心目标**。为了应对不同类型的分享（如：单主题技术剖析、多子主题解析、或方法论宣讲），PPT 结构**必须且只能**遵循一个高度统一的**线性叙事 DSL**。

## 🎯 核心架构原则

1. **线性平铺结构 (Linear Flattening)**
   - **绝对禁止使用嵌套目录** (例如 `02.concepts/`)，因为 PPT 本质是线性展示。所有幻灯片文件必须直接平铺在项目下。

2. **子主题拆分规则 (Sub-topic Splitting)**
   - 当一个核心目标涉及多个关键概念时（如 Prompt + Context Engineering、MCP + Skill），允许在“概念层 (Concept)”进行拆分。
   - 文件命名必须使用子序号：`02.1.xxx.md`, `02.2.xxx.md`, `02.3.xxx.md`。
   - **约束**：所有子主题必须服务于同一个核心目标，且建议数量 $\le$ 3。

3. **收敛叙事法则 (Narrative Convergence)**
   - 多子主题 PPT 必须遵守：**先统一提出问题 $\rightarrow$ 再拆分核心概念 $\rightarrow$ 最后重新统一应用与实践**。否则听众会觉得是两个拼接的分享。

---

## 🪨 内页结构铁律 (Strict Page Constraints)

除了宏观架构外，Agent 必须强制遵守以下页面内部组件约束（参考 Vibe Coding 模板）：

1. **总览页特权 (`01.overview.md`)**:
   - 必须以 `# 课程目标` 开头，提炼本次分享能解决的 1~3 个核心问题（建议配合 `v-mark` 圈设重点）。
   - 紧接着必须是 `# 课程大纲`，罗列将要讲解的目录结构。
2. **章节收尾雷打不动 (Chapter Summary)**:
   - **所有的 Markdown 文件**（从 `01` 到最后一个实战文件，不含 QA），其生成内容的最后一页幻灯片，必须是 `# 本章小结：xxxx` 或者是 `🎯 核心收获` 模块。
   - 用 1~3 句话高度概括当前文件所讲的核心增量信息。让听众在进入下一目标前完成记忆固化。

## 🏗 核心架构分流 (Architecture Divergence)

PPT 目录树**禁止使用任何文件夹嵌套**，必须全部平铺在根目录。
所有的 PPT 项目都由一个 `slides.md` 作为总入口（用于汇编所有碎片文件并配置主题）。
根据你要讲解的核心目标，必须从以下**两种严格的树状模型**中选择其一：

### 模式 A：单一主题的标准 5 层结构 (Standard Linear Flow)
**适用场景**：只聚焦于一个核心概念或工具（如 Vibe Coding 方法论、单一框架深度剖析）。
**结构要求**：原理探讨与核心解法分属于独立的幻灯片层级（即固定 `01` 到 `05` 必定只有 5 个文件）。

```text
your-ppt/
├── slides.md               # 汇总入口，根据实际文件名更新其中的 src 路径
├── 01.overview.md          # [提出问题] 痛点在哪？定义现实与理想的差距，定调核心目标。
├── 02.principle.md         # [拆解概念] 剥开表象看本质。讲解底层的核心机制或理论基石。
├── 03.solution.md          # [核心解法] 给出解决该问题的标准打法体系或撒手锏功能 (可语义化为 features 等)。
├── 04.practice.md          # [实战落地] Show me the code. 具体的代码实现或实例演示。
└── 05.QA.md                # [总结折衷] 优缺点与代价 (Pros/Cons)，延伸进阶阅读与答疑。
```

---

### 模式 B：多子主题的“微结构 + 收敛”模型 (Micro-Structure & Convergence)
**适用场景**：核心目标由多个独立但关联的子主题拼图构成（如：Prompt + Context Engineering, 或 MCP + Skill）。
**结构要求**：坚守宏观的 5 个固定槽位！绝不允许随意顺延导致 `04.practice` 漂移。把概念层 (`02`) 进行**水平切分**为多个微结构，在应用层 (`03`) 强制进行大融合。

```text
your-ppt/
├── slides.md               # 汇总入口，根据实际生成的文件名更新各个 src 路径，确保线性流
├── 01.overview.md          # [统一抛出问题] 为什么会有痛点？引出我们亟需综合手段。
├── 02.1.topic-a.md         # [微结构 A] 讲解 A 的核心概念与 A 的专属用法。
├── 02.2.topic-b.md         # [微结构 B] 讲解 B 的核心概念与 B 的专属用法。
├── 03.integration.md       # [强制收敛] (必选!) A 和 B 在真实业务流中是如何组合联动的？产生 1+1>2 的化学反应。
├── 04.practice.md          # [综合实战] 围绕收敛后的组合方案，提供端到端的综合实战演示。
└── 05.QA.md                # [总结折衷] 整体方案的代价、折衷与问答。
```

## ✅ 分享大纲质量终检 (Self-Review Checklist)

在 Agent 生成所有的大纲后，在最终输出前，**必须执行**以下自检：

### 架构与配置层面检查

- [ ] (关键红线) 所有的 Markdown 文件是否**完全平铺在根目录 (零嵌套)**？
- [ ] (关键红线) 是否坚守了 **5个固定的宏观槽位 (01~05)**，如果有多个并行的子主题，是否正确使用了**水平拆分 (02.1, 02.2)** 而不是破坏了整体序号顺延？
- [ ] `slides.md` 里的 `src:` 路径是否**准确匹配**你实际生成的 `02.xxx.md` 和 `03.xxx.md` 文件名？
- [ ] 首个章节 (`01.x`) 有没有以 `# 课程目标` 和 `# 课程大纲` 开头，并清晰地牵引出痛点？
- [ ] 每个章节 (`01`~最后一页实战) 幻灯片末尾，是否采用 `template-content.md` 中预设的 `<div class="p-4...">` HTML 样式输出了 `🎯 核心收获` 模块？
- [ ] 多子主题结构中，是否包含了 **强耦合的收敛节点 (`03.integration` 等)** 以完成闭环？
- [ ] `05.QA` 中有没有指明优缺点 (Pros/Cons) 与整体业务代价折衷？
- [ ] 有没有补充进阶的文档资料 (References)？

### Slidev 高级特性检查

- [ ] 是否在关键页面添加了 **Speaker Notes** (`<!-- 演讲者备注：... -->`)？
- [ ] 是否**彻底摒弃了复杂的 `div` 嵌套**，而是全面采用了原生 Markdown 及 Comark 纯净语法 (如 `!!steps` 等)，而非单纯堆砌 HTML？
- [ ] 对于复杂的逻辑/流程图表，是否提取为了具有高度复用性的 Vue 组件 (`components/*.vue`)，并配合全屏自定义 Layout (例如 `layout: full-vibe`，且指定了交互需要的 `clicks` 参数) 进行展示？
- [ ] 在演示源码或进行技术级解读时，是否使用了**渐进式代码步进**语法 (例如 ````ts {1|2-5|all}````)？
- [ ] 在 `04.practice` 展示重构/修复前后的代码对比时，是否使用了 **Magic Move** 特权实现震撼转场？
- [ ] 重要的实战代码片段，是否按需启用了 `{monaco}` 以便现场可复现修改？
- [ ] 是否使用了**多样化的布局** (cover, two-cols, center, etc.) 而非单一布局？
- [ ] 重要概念是否使用 **v-mark 高亮** (underline, circle 等) 进行标注？
