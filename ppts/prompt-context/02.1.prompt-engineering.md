# Prompt 工程的三个进化阶段

Zero-Shot · Few-Shot · Chain-of-Thought

---

# Zero-Shot → Few-Shot → Chain-of-Thought

<v-clicks>

**Zero-Shot（零样本）**：仅有任务指令，无示例
- 适用：简单、明确的任务（翻译、格式化、分类）
- 风险：复杂任务下，模型自由发挥，输出不稳定

**Few-Shot（少样本）**：提供 2-5 个示例，引导输出模式
- 引用：Brown et al. 2020, *Language Models are Few-Shot Learners*, NeurIPS 2020
- 关键：**示例质量 >> 示例数量**，噪声示例比无示例更糟

**Chain-of-Thought（思维链）**：显式推理过程，再给结论
- 引用：Wei et al. 2022, *Chain-of-Thought Prompting Elicits Reasoning*, NeurIPS 2022
- 核心咒语：`"Let's think step by step"`
- 效果：复杂推理准确率提升 **40-60%**

</v-clicks>

---

# CoT 2.0：Extended Thinking

<v-clicks>

传统 CoT：推理过程混入输出流，**不可折叠**

2024-2025进化：**模型原生推理层**，推理Token独立于输出

</v-clicks>

<v-click>

| 对比 | 传统 CoT | Extended Thinking |
|------|---------|------------------|
| 推理可见性 | 输出流中可见 | 独立思考 Token，可折叠 |
| 控制粒度 | Prompt 引导 | `thinking_budget` 参数控制 |
| 代表模型 | GPT-4 | Claude 3.7 Sonnet、o1、o3 |
| 适用场景 | 通用推理 | 数学/代码/复杂分析 |

</v-click>

<v-click>

```python
# Claude 3.7 Extended Thinking
response = client.messages.create(
    model="claude-sonnet-4-5",
    thinking={"type": "enabled", "budget_tokens": 10000},
    messages=[{"role": "user", "content": "分析这段代码的竞态条件..."}]
)
```

</v-click>

# Structured Output：输出格式约束

<v-clicks>

**问题**：AI 自然语言输出，程序无法直接解析

**解法**：强制结构化输出格式（JSON Schema / Tool Use）

</v-clicks>

<v-click>

```
❌ 无格式约束：
"这段代码有三个问题：第一，..."  → 需额外解析逻辑

✅ 约束输出格式：
{
  "issues": [
    {"type": "security", "line": 42, "severity": "high", "desc": "SQL injection"},
    {"type": "perf",     "line": 87, "severity": "med",  "desc": "N+1 query"}
  ],
  "summary": "发现 2 个问题，建议立即修复"
}
```

</v-click>

<v-click>

**实现路径**：OpenAI JSON Mode / Anthropic Tool Use / Zod Schema 约束

> **规则**：输出若被代码消费，必须约束格式。

</v-click>

# Prompt 反例：三个真实失败场景

> 来自真实项目日志记录。

::left::

<div class="col-bad">

### ❌ 场景 1：模糊指令

```
"帮我优化下代码"
```

- 问题：AI 不知道优化目标（性能？可读性？体积？）
- 结果：随机改动，无法验证

### ❌ 场景 2：缺少约束

```
"写一个排序函数"
```

- 问题：无语言、无数据类型、无复杂度要求
- 结果：冒泡排序，10 行 TypeScript，无类型注解

</div>

::right::

<div class="col-good">

### ✅ 场景 1：精准指令

```
"用 TypeScript 实现时间复杂度 O(n log n) 的
整数数组排序，禁止使用 any"
```

### ✅ 场景 2：完整约束

```
"用 TypeScript 实现快速排序：
- 输入：number[]
- 输出：number[]
- TypeScript strict mode，每行必须有类型标注"
```

</div>

# .claudeprompt：团队级 Prompt 版本化管理

<v-clicks>

**核心理念**：Prompt 是代码，应纳入版本控制。

**问题**：散落各处的 Prompt（Slack / 聊天记录 / 本地文件）→ 版本混乱，无法回溯

**解法**：仓库化管理 + MR 审核 + CI 验证

</v-clicks>

<v-click>

```
项目结构：

prompts/
├── system/
│   ├── code-review.md          # 代码审查助手
│   ├── vue3-generator.md       # Vue3 组件生成
│   └── bug-analyzer.md         # Bug 分析
├── templates/
│   ├── pr-description.md
│   └── test-case-gen.md
└── CHANGELOG.md               # 版本变更记录
```

</v-click>

<v-click>

> Git 追踪变更原因 + AB 测试验证效果 + 团队共享同一版本

</v-click>

# ICIO 框架：Prompt 的四个组成部分

<v-clicks>

**I — Instruction**：执行什么任务
**C — Context**：需要的背景知识
**I — Input**：需要处理的数据
**O — Output**：期望的输出形态

</v-clicks>

<v-click>

```
┌─────────────┬──────────────────────────┬──────────┐
│ 组成部分     │ 作用                      │ 必须     │
├─────────────┼──────────────────────────┼──────────┤
│ Instruction │ 告诉模型执行什么任务       │ ✅ 必须  │
│ Context     │ 补充背景，引导理解         │ 推荐     │
│ Input Data  │ 需要处理的具体内容         │ 视任务   │
│ Output      │ 格式、长度、语言要求       │ 推荐     │
└─────────────┴──────────────────────────┴──────────┘
```

</v-click>

[Context]
- TypeScript strict mode
- 高并发场景（~1000 req/s）
- 团队规范：错误必须 throw

[Input]
  async function getData(id) {
    try {
      const res = await fetch(`/api/data/${id}`)

    } catch(e) {}  // ← 问题
  }

[Output]
JSON 格式：issues、severity、suggestion
