# ICIO 框架：Prompt 的四个组成部分

<v-clicks>

**I — Instruction（指令）**：你要 AI 做什么  
**C — Context（上下文）**：AI 需要知道的背景  
**I — Input（输入数据）**：AI 需要处理的内容  
**O — Output（输出格式）**：你期望的输出形态

</v-clicks>

<v-click>

```
┌─────────────┬──────────────────────────┬──────────┐
│ 组成部分     │ 作用                      │ 必须     │
├─────────────┼──────────────────────────┼──────────┤
│ Instruction │ 告诉模型执行什么任务       │ ✅ 必须  │
│ Context     │ 补充背景，引导理解         │ 推荐     │
│ Input Data  │ 需要处理的具体内容         │ 视任务   │
│ Output      │ 格式、长度、语言要求       │ 推荐     │
└─────────────┴──────────────────────────┴──────────┘
```

</v-click>

# ICIO Before vs After

**❌ Before（缺少框架）：**

```
帮我审查一下这个函数
```

<v-click>

**✅ After（ICIO 框架）：**

```
[Instruction]
请对以下 TypeScript 函数进行代码审查。

[Context]
- 项目使用 TypeScript strict mode，禁止 any 类型
- 函数在高并发场景下调用（每秒约 1000 次请求）
- 团队规范：错误必须明确 throw，不能静默吞掉

[Input]
  async function getData(id) {
    try {
      const res = await fetch(`/api/data/${id}`)
      return res.json()
    } catch(e) {}  // 问题在这里
  }

[Output]
请以 JSON 格式输出：issues（问题列表）、severity（严重程度）、suggestion（修复建议）
```

</v-click>

<v-click>

> **框架不是束缚，是让 AI 稳定输出的结构化合同。**

</v-click>

# .cursorrules 实战示例

<style>
pre { font-size: 11px !important; line-height: 1.4; }
</style>

```yaml
# .cursorrules — Vue3 项目规范
---
description: Vue3 + TypeScript 项目开发规范
globs: ["src/**/*.vue", "src/**/*.ts"]
---

## 技术栈
- Vue 3 Composition API（禁止 Options API）
- TypeScript strict mode（禁止 any 类型，禁止非空断言 !）
- Pinia 状态管理（禁止 Vuex）
- Vitest 单元测试（每个业务函数必须有测试用例）

## 组件规范
- 组件文件名：PascalCase（BaseButton.vue）
- Props 必须有 TypeScript 类型定义
- emit 必须使用 defineEmits<{...}>() 明确声明
- 禁止 defineComponent，直接使用 <script setup>

## 代码风格
- 异步函数必须有 try/catch，错误必须 log 并 throw，不能静默吞掉
- API 调用统一通过 src/api/ 目录，禁止在组件中直接 fetch
```

# System Prompt 团队级模板

<style>
pre { font-size: 11px !important; line-height: 1.4; }
</style>

<v-clicks>

适用于：API 调用、自建工具、团队共享助手

</v-clicks>

<v-click>

```markdown
你是 [团队名称] 的高级前端工程师助手。

## 角色定义
- 精通 Vue3 Composition API、TypeScript、Pinia
- 了解团队代码规范（参见附件）
- 代码审查标准：安全性 > 性能 > 可读性 > 简洁性

## 工作原则
1. 优先给出可直接使用的代码，而不是解释性文字
2. 发现潜在问题时，主动指出，不要只回答被问到的部分
3. 输出代码必须包含 TypeScript 类型，禁止 any
4. 涉及数据库操作，必须提醒事务和回滚处理

## 输出格式
- 代码审查：JSON 格式（issues 数组 + summary）
- 代码生成：直接给出代码，在注释中说明关键决策
- 架构建议：先给结论，再给理由

## 禁止行为
- 不要假设用户知道最佳实践，明确说明原因
- 不要生成任何包含 console.log 的生产代码
```

</v-click>

# 精准投喂：Context 的外科手术原则

**核心理念**：Context 是精准手术刀，不是散弹枪。

<v-clicks>

**相关性过滤**
- 只提供与当前任务直接相关的文件/代码片段
- 宁可分多次问，不要把整个代码库堆进去

**结构化输入**
- 使用 XML 标签或 Markdown 分隔不同信息块
- 让 AI 清楚哪部分是背景、哪部分是数据、哪部分是问题

**位置即权重**
- 最重要的约束 → System Prompt（开头）
- 当前问题的关键补充 → 用户输入（结尾）
- 参考资料 → 中间（不要放关键约束）

</v-clicks>

# 结构化输入示例

<v-clicks>

**❌ 混乱的输入：**

```
帮我看看这段代码，它是我们项目里的数据处理逻辑，
这个项目用的是 TypeScript，我们要求严格类型，
下面是代码：async function process(data) {...}，
主要问题是有时候会 crash
```

</v-clicks>

<v-click>

**✅ 结构化的输入：**

```xml
<context>
项目：nn-client-all | 技术栈：TypeScript strict | Node.js 18
问题现象：每天凌晨 2 点定时任务偶发 crash，无明确报错
</context>

<code>
async function process(data: unknown) {
  const result = (data as any).items.map(...)  // 出问题的行
}
</code>

<question>
请分析 crash 的根本原因，并给出类型安全的修复方案
</question>
```

</v-click>

# 典型 Anti-Pattern

<style>
pre { font-size: 12px !important; line-height: 1.4; }
li { font-size: 0.88rem !important; }
</style>

<v-clicks>

**🚫 Anti-Pattern 1：把整个代码库扔给 AI**
```
@整个src目录 帮我找一下性能问题
```
→ Context 超限，AI 开始随机猜测，答非所问

**🚫 Anti-Pattern 2：System Prompt 与 User Prompt 冲突**
```
System: 用中文回复
User:   Please respond in English
```
→ 行为不可预期，模型在两个指令间摇摆

**🚫 Anti-Pattern 3：无格式约束**
```
分析这段日志，告诉我有哪些错误
```
→ 输出是自然语言，程序无法解析，需要再次处理

**🚫 Anti-Pattern 4：把 API Key 放进 Context**
```
用这个 key：sk-xxx... 调用接口
```
→ Key 会进入对话历史，存在泄露风险

</v-clicks>

# Prompt 版本管理

<v-clicks>

**核心理念**：Prompt 是代码，应该像管理代码一样管理 Prompt。

</v-clicks>

<v-click>

```
项目结构示例：

prompts/
├── system/
│   ├── code-review.md          # 代码审查助手 System Prompt
│   ├── vue3-generator.md       # Vue3 组件生成 System Prompt
│   └── bug-analyzer.md         # Bug 分析助手 System Prompt
├── templates/
│   ├── pr-description.md       # PR 描述模板
│   └── test-case-gen.md        # 测试用例生成模板
└── CHANGELOG.md                # Prompt 版本变更记录
```

</v-click>

<v-clicks>

- **Git 追踪**：每次 Prompt 修改，记录修改原因和效果对比
- **AB 测试**：关键 Prompt 变更前，用固定测试用例验证效果
- **团队共享**：Prompt 是团队资产，放进仓库，不锁在个人电脑

</v-clicks>

# 最佳实践总结

<v-clicks>

**结构层面（ICIO）**
- 所有非trivial的任务，都用 ICIO 框架组织 Prompt
- Output 部分明确格式要求，优先 JSON

**工程化层面（全局约束）**
- 项目规范 → `.cursorrules` / `CLAUDE.md`
- 团队级助手 → System Prompt 模板化 + 版本管理

**Context 管理层面（精准投喂）**
- 关键约束 → System Prompt（开头）
- 当前问题 → 用户输入（结尾）
- 无关内容 → 坚决过滤，不要"以防万一"

</v-clicks>
