# 2. Context 工程

如何喂得准

---

# 什么是 Context？

<v-clicks>

**一句话定义**：Context = 模型本次推理时**实际看到的全部输入 Token**。

不只是你写的那段话——它包括：

</v-clicks>

<v-click>

```
Context = System Prompt
        + Memory 注入（CLAUDE.md / `.cursorrules` 或 `.cursor/rules/`）
        + RAG 召回内容  ← 向量检索：把相关文档找出来注入窗口
        + 对话历史
        + 工具调用结果
        + 当前 User 输入（你写的 Prompt）
```

</v-click>

<v-click>

> **关键认知**：Context 不是"你想让 AI 知道的"，而是"AI 此刻确实看见的"。
> 没进窗口的，等于不存在。

</v-click>

---

# 为什么需要 Context Engineering？

<v-clicks>

**三个硬约束**逼着你做 Context 工程：

</v-clicks>

<v-click>

| 约束 | 含义 | 工程后果 |
|------|------|---------|
| **窗口有上限** | 主流最大 1M Token，大部分 200K Token | 塞不下整个代码库 |
| **Token 有成本** | 输入 $3/M、输出 $15/M | 盲目堆料 → 成本爆炸 |
| **注意力会稀释** | Lost in the Middle 现象 | 信息越多 → 关键越被淹没 |

</v-click>

<v-click>

**Context Engineering 的目标**：

| 把…… | 翻译为…… |
|------|---------|
| 散乱信息（聊天碎片 / 整目录） | **结构化窗口**（System / 历史 / 输入分层） |
| 全量灌入（越多越好） | **精准切片**（最小有效集） |
| 一次性投喂 | **分级注入**（固定层 / 动态层 / 实时层） |

</v-click>

<v-click>

> Context 工程不是"塞东西"，是**有限窗口里的外科手术**。

</v-click>

# Context Window 的完整结构

```
┌─────────────────────────────────────────────────────────────────┐
│                         Context Window                          │
│                                                                 │
│  ┌──────────────────┐  ← 全局规则、角色定义、约束（CLAUDE.md）│
│  │  System Prompt   │  (固定，高权重)                          │
│  └──────────────────┘                                           │
│  ┌──────────────────┐  ← 外挂知识库，按相关性检索注入           │
│  │   RAG 召回内容   │  (动态注入)                              │
│  └──────────────────┘                                           │
│  ┌──────────────────┐  ← 短期记忆，越长越占 Token             │
│  │   对话历史        │  (滚动窗口)                              │
│  └──────────────────┘                                           │
│  ┌──────────────────┐  ← 实际问题 + @引用文件                   │
│  │   当前 User 输入 │  (高权重)                                │
│  └──────────────────┘                                           │
└─────────────────────────────────────────────────────────────────┘
```

<v-click>

> **四层分工明确**：固定规则 / 召回知识 / 滚动历史 / 当前任务。每一层有自己的工程手法。

</v-click>

# 首尾定律：Attention 是位置敏感的

> 引用：[Liu et al. 2023, *Lost in the Middle*](https://arxiv.org/abs/2307.03372)

<v-clicks>

**研究发现**：文档位置直接影响模型注意力权重——

</v-clicks>

<v-click>

```
注意力权重分布：

  ▲
  │ ██                                              ██
  │ ███                                            ███
  │ ████                                          ████
  │ █████                                        █████
  │ ██████                                      ██████
  │ ████████                                  ████████
  │ █████████████                        █████████████
  └──────────────────────────────────────────────────►
       开头（高）          中间（低）         结尾（高）
```

</v-click>

<v-clicks>

- 20 文档检索答案：**第 1 位 71%** 准确率，**第 10 位 51%**（[Liu et al. 2023](https://arxiv.org/abs/2307.03372)）
- **首尾定律**：关键约束 → System Prompt（开头）；当前任务 → User Input（结尾）
- 中间内容会被"淹没"——**无关信息坚决过滤**

</v-clicks>

<v-click>

> **工程原则**：宁可短而精，不要长而散；位置即权重。

</v-click>

# Token 经济：算清你的账

<v-clicks>

**Token 计算基准**：
- 1 个中文字符 ≈ **2 Token**
- 1 个英文单词 ≈ **1.3 Token**
- 1 行代码（平均）≈ **6 Token**

</v-clicks>

<v-click>

**主流 LLM 定价参考（示例）**：输入约 $3/M Token · 输出约 $15/M Token（因模型和厂商而异）

</v-click>

<v-click>

```
团队规模算账：

100 个开发 × 100 次/天 × 平均 800 Token/次
= 8,000,000 Token/天
= 2.4 亿 Token/月
≈ $720/月（仅输入，未含输出）
```

</v-click>

<v-click>

> Context 不是免费午餐——**每塞一个 Token，都在花钱 + 稀释注意力**。

</v-click>

# 选型决策：RAG vs 长上下文 vs 全量

<v-clicks>

**按场景选择，而非"窗口越大越好"**：

</v-clicks>

<v-click>

```
你的场景？
│
├─ 答案在明确文档，接受秒级延迟
│   └─ → RAG（向量检索 + 召回注入）
│
├─ 需跨 200+ 文档推理，无明确"正确答案"
│   └─ → 长上下文（注意首尾定律）
│
├─ 代码库 < 50K Token，可全部放入
│   └─ → 全量放入（关键约束放 System / 结尾）
│
└─ 频繁交互，每次只问一小部分
    └─ → 动态切块 + 滚动窗口
```

</v-click>

<v-click>

> **窗口大 ≠ 效果好**——attention 优化与组织策略才是关键。

</v-click>

# Context 压缩：三种工程手法

<v-clicks>

**摘要**：保留语义核心，去除冗余表述

**索引**：告诉 AI"答案在哪找"，而非直接给答案（让它自己 read）

**分级注入**：System（固定）/ 对话（动态）/ Tool（实时）—— 各取所需

</v-clicks>

<v-click>

```python
# 分级注入示例
messages = [
    {"role": "system",      "content": "你是一个 Vue3 审查助手"},
    {"role": "user",        "content": "分析这个组件的问题"},
]
# Tool Result 运行时注入（按需）
tool_results = [{"content": summarize(file_content), "source": "Login.vue"}]
```

</v-click>

<v-click>

> 一个高级技巧：**让 AI 自己决定读什么**——给它工具，比预先塞所有文件更省 Token。

</v-click>

# Context 反模式：好 vs 坏

::left::

<div class="col-bad">

### ❌ 散弹枪式投喂

```
帮我看看这段代码，它是我们项目里
的数据处理逻辑，这个项目用的是
TypeScript，我们要求严格类型，
下面是代码：async function
process(data) {...}，主要问题
是有时候会 crash
```

- 背景 / 数据 / 约束混成一团
- AI 答非所问，假设一堆不存在条件

</div>

::right::

<div class="col-good">

### ✅ 精准外科手术

```xml
<context>
项目：nn-client-all | TS strict | Node 18
问题现象：每天凌晨 2 点偶发 crash
</context>

<code>
async function process(data: unknown) {
  const result = (data as any).items.map(...)
}
</code>

<question>
分析 crash 根本原因，给出类型安全
的修复方案
</question>
```

- 背景 / 代码 / 问题分块清晰
- 用 XML/Markdown 结构化标记

</div>

# Context 安全红线

<v-clicks>

**Context 不只是性能问题，更是攻击面**——四个红线必须知道：

</v-clicks>

<v-click>

| 风险 | 攻击形态 | 防御 |
|------|---------|------|
| **数据泄漏** | `.env`、密钥、内部文档被读入 Context → 模型复述/记入历史 | `.cursorignore` / `.aiignore` 黑名单 + 提交前扫描 |
| **Context 污染** | 过期文档、错误注释混入 → AI 误判 | 仓库即文档 + 定期清理 + 版本标记 |
| **Token 失控** | 盲目堆 @全目录 → 成本爆炸 + Lost in the Middle | 最小有效集原则 + 监控 Token 消耗 |
| **跨会话泄漏** | A 项目的 Context 被 B 项目摘要带过去 | 项目隔离 / 上线前敏感信息 mask |

</v-click>

<v-click>

```
✅ 三道防线：
   ① 入口拦截：.cursorignore / git-secrets / 提交前扫描
   ② 分级权限：System Prompt 声明红线（"绝不读取 /configs/prod"）
   ③ 出口审计：所有 Context 注入留痕，便于追溯泄漏
```

</v-click>

<v-click>

> **核心原则**：**没进窗口 = 不存在**，反过来也成立——**进了窗口 = 不可撤回**。

</v-click>

# 工具复用：不同工具的 Context 注入形态

<v-clicks>

**Context 注入不是手动拼字符串**——主流工具各有自己的"注入语法"：

</v-clicks>

<v-click>

| 工具 | Context 注入语法 | 自动注入源 |
|------|----------------|----------|
| **Cursor** | `@file` `@code` `@docs` `@web` `@git` | `.cursorrules` 或 `.cursor/rules/` |
| **Claude Code** | `@path` + `Read` tool + Skill | `CLAUDE.md` + `~/.claude/CLAUDE.md` |
| **OpenCode** | `@path` + 自定义 commands | `AGENTS.md` |
| **通用 RAG** | 向量检索 → 相似度 Top-K 召回 | 知识库索引 |
| **通用 MCP**（Model Context Protocol）| 标准化协议 → 可插拔 Context 源 | MCP Server 配置 |

</v-click>

<v-click>

```
工程实践三原则：
① 优先用工具原生注入语法，不要手抄文件内容
② 项目级规范放 `.cursorrules`（或新版 `.cursor/rules/`）/ CLAUDE.md / AGENTS.md，进 Git
③ 跨工具知识库走 MCP——一处配置，多端复用
```

</v-click>

<v-click>

> **核心**：让 Context 注入变成**声明式配置**，而不是**手工搬运**。

</v-click>

# 章节小结

<v-clicks>

✅ **定义**：Context = 模型本次实际看到的全部 Token；没进窗口 = 不存在

✅ **三大约束**：窗口有上限 / Token 有成本 / 注意力会稀释 → 必须做工程化

✅ **核心机制**：首尾定律（Liu 2023）—— 关键约束放 System，当前任务放 User Input

✅ **安全**：数据泄漏 / Context 污染 / Token 失控 / 跨会话泄漏——**进了窗口 = 不可撤回**

✅ **复用**：声明式注入（`.cursorrules` 或 `.cursor/rules/` / `CLAUDE.md` / MCP）> 手工搬运

</v-clicks>
