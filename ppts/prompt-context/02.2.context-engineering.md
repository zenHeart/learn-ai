# Context Window 的完整结构

```
┌─────────────────────────────────────────────────────────────────┐
│                         Context Window                          │
│                                                                 │
│  ┌──────────────────┐                                           │
│  │  System Prompt   │  ← 全局规则、角色定义、约束（CLAUDE.md）  │
│  │  (固定，高权重)   │                                           │
│  └──────────────────┘                                           │
│  ┌──────────────────┐                                           │
│  │   RAG 召回内容   │  ← 外挂知识库，按相关性检索注入             │
│  │  (动态注入)      │                                           │
│  └──────────────────┘                                           │
│  ┌──────────────────┐                                           │
│  │   对话历史        │  ← 短期记忆，越长越占 Token               │
│  │  (滚动窗口)      │                                           │
│  └──────────────────┘                                           │
│  ┌──────────────────┐                                           │
│  │   当前 User 输入 │  ← 实际问题 + @引用文件                   │
│  │  (高权重)        │                                           │
│  └──────────────────┘                                           │
└─────────────────────────────────────────────────────────────────┘
```

# Token 真实代价

<v-clicks>

**Token 计算基准**：
- 1 个中文字符 ≈ **2 Token**
- 1 个英文单词 ≈ **1.3 Token**
- 1 行代码（平均）≈ **6 Token**

</v-clicks>

<v-click>

**Claude Sonnet 4.5 定价**：
- 输入：$3 / 百万 Token
- 输出：$15 / 百万 Token

</v-click>

<v-click>

**现场算账**：

```
100 个开发 × 100 次/天 × 平均 800 Token/次
= 8,000,000 Token/天
= 2.4 亿 Token/月
≈ $720/月（仅输入）
```

</v-click>

<v-click>

# Lost in the Middle Problem

> 引用：Liu et al. 2023, *Lost in the Middle: How Language Models Use Long Contexts*, arXiv 2024

<v-clicks>

**研究发现**：文档位置影响模型注意力权重

</v-clicks>

<v-click>

```
注意力权重分布：

  ▲
  │ ██                                              ██
  │ ███                                            ███
  │ ████                                          ████
  │ █████                                        █████
  │ ██████                                      ██████
  │ ████████                                  ████████
  │ █████████████                        █████████████
  └──────────────────────────────────────────────────►
       开头（高）          中间（低）         结尾（高）
```

</v-click>

<v-clicks>

- 20 文档中检索答案：位于**第 1 位** → 准确率 **71%**，位于**第 10 位** → 下降至 **51%**
- **首尾定律**：关键信息放开头（System Prompt）或结尾（User Input）
- 中间内容会被"淹没"，无关信息坚决过滤

</v-clicks>

---

# 首尾定律：Attention 机制的本质

<v-clicks>

LLM 的注意力机制对**位置**敏感：

- **System Prompt**（开头）：高权重，全局规则适用
- **User Input**（结尾）：高权重，当前任务相关
- **中间内容**：低权重，容易被忽略

</v-clicks>

<v-click>

> **工程原则**：关键约束 → System Prompt；当前问题 → User Input。

</v-click>

# Context 投喂：精准手术 vs 散弹枪

::left::

<div class="col-bad">

### ❌ 散弹枪式投喂

```
帮我看看这段代码，它是我们项目里的数据处理逻辑，
这个项目用的是 TypeScript，我们要求严格类型，
下面是代码：async function process(data) {...}，
主要问题是有时候会 crash
```

- 问题：背景 / 数据 / 约束混在一起
- 结果：AI 答非所问，假设一堆不存在条件

</div>

::right::

<div class="col-good">

### ✅ 精准投喂

```xml
<context>
项目：nn-client-all | TypeScript strict | Node.js 18
问题现象：每天凌晨 2 点偶发 crash
</context>

<code>
async function process(data: unknown) {
  const result = (data as any).items.map(...)
}
</code>

<question>
分析 crash 根本原因，给出类型安全的修复方案
</question>
```

</div>

# RAG vs 长上下文：选型决策树

<v-clicks>

**按场景选择，而非越大越好**：

</v-clicks>

<v-click>

```
你的场景？
│
├─ 答案在明确文档，接受分钟级延迟
│   └─ → RAG（检索增强生成）
│
├─ 需跨 200+ 文档推理，无明确"正确答案"
│   └─ → 长上下文（注意首尾定律）
│
├─ 代码库 < 50K Token，可全部放入
│   └─ → 全量放入，注意位置（System/结尾）
│
└─ 频繁交互，每次只问一小部分
    └─ → 动态切块 + 滚动窗口
```

</v-click>

# Context 压缩：工程手法

<v-clicks>

**摘要**：保留语义核心，去除冗余

**索引**：告诉 AI "答案在哪找"，而非直接给答案

**分级注入**：System（固定）/ 对话（动态）/ Tool（实时）

</v-clicks>

<v-click>

```python
# 分级注入示例
messages = [
    {"role": "system",      "content": "你是一个 Vue3 审查助手"},
    {"role": "user",        "content": "分析这个组件的问题"},
]
# Tool Result 运行时注入
tool_results = [{"content": summarize(file_content), "source": "Login.vue"}]
```

</v-click>
