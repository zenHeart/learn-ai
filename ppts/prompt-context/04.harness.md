# 六大 Harness 原则

<style>
table { font-size: 0.82rem !important; }
td, th { padding: 0.4rem 0.6rem !important; }
</style>

| # | 原则 | 核心理念 | 类比 |
|---|------|---------|------|
| P1 | **仓库即记录系统** | Agent 看不到的等于不存在 | 代码 > Slack 消息 |
| P2 | **地图而非手册** | ~100行 AGENTS.md 指向深层 | 目录 > 百科全书 |
| P3 | **机械化强制执行** | lint 规则 > 文档约定 | 编译错误 > 代码审查意见 |
| P4 | **面向 Agent 的可读性** | 让 AI 能读 UI、日志、指标 | 可观测性 > 注释 |
| P5 | **吞吐量改变合并哲学** | 纠错成本 < 等待成本 | 短生命周期 PR |
| P6 | **熵管理即垃圾回收** | 编码"黄金原则" + 周期清理 | GC > 手工维护 |

<v-click>

> 这 6 个原则不是独立的——它们共同构成一个**自持续、自演化**的 AI 工程生态。

</v-click>

# P3：机械化强制执行

> <span v-mark.underline.yellow="1">文件会腐烂；lint 规则不会。</span>

::left::

### 什么应该成为 lint 规则

<v-clicks>

- 架构层依赖方向（不允许 UI → Repo 直接调用）
- 文件大小限制（超过 N 行触发警告）
- 命名约定（文件名 PascalCase，API 函数 camelCase）
- 日志格式（禁止 `console.log`，必须结构化日志）
- 平台可靠性要求（关键路径响应 < 800ms）

</v-clicks>

::right::

### 实战：自定义 ESLint 规则

<v-click>

```javascript
// eslint-rules/no-direct-fetch.js
// 禁止在组件中直接调用 fetch，必须走 src/api/
module.exports = {
  create(context) {
    return {
      CallExpression(node) {
        if (node.callee.name === 'fetch') {
          context.report({
            node,
            // 错误信息直接嵌入修复指令（供 Agent 读取）
            message: '[Harness] 禁止直接 fetch。' +
              '请使用 src/api/ 目录下的封装函数。' +
              '参考: docs/FRONTEND.md#api-calls'
          });
        }
      }
    };
  }
};
```

</v-click>

# P4：面向 Agent 的可读性

> Agent 能直接推理的，才是真正的 Context。

::left::

### ❌ Agent 无法读取

<v-clicks>

- Slack 讨论中的架构决策
- Google Docs 里的产品规范
- 工程师头脑中的"不成文规定"

# P5 + P6：吞吐量与熵管理

::left::

### P5：吞吐量改变合并哲学

<v-clicks>

**传统工程**
- 严格的 PR 审查门控
- 等待 → 修改 → 等待
- 错误代价高 → 保守

**Harness 工程**
- PR 生命周期极短
- 纠错成本 < 等待成本
- Agent 出错 → 后续 PR 修复

</v-clicks>

<v-click>

> 在低吞吐量环境里，等待是负责任的。  
> 在这里，等待才是风险。

</v-click>

::right::

### P6：熵管理即垃圾回收

<v-clicks>

**问题**：Agent 会复制仓库中已有的模式——包括坏的模式。

**解法**：编码"黄金原则" + 周期性清理 Agent

</v-clicks>

<v-click>

```yaml
# 每周自动运行的清理 Agent
schedule: "0 2 * * 0"  # 每周日凌晨 2 点
tasks:
  - scan_stale_docs        # 扫描过期文档
  - find_pattern_drift     # 检测模式漂移
  - update_quality_scores  # 更新质量评分
  - open_refactor_prs      # 发起重构 PR
```

</v-click>

<v-click>

> 技术债像高息贷款——定期小额偿还，  
> 比积累后一次性大痛苦划算得多。

</v-click>

# Ralph 循环：Agent 工作流的引擎

```
                    ┌──────────────────────────────────────────┐
                    │            Ralph Loop                     │
                    │                                          │
  人类输入需求  ──▶  │  1. 新开会话（新鲜上下文）               │
                    │         ↓                                │
                    │  2. Agent 读取 AGENTS.md（地图）          │
                    │         ↓                                │
                    │  3. 执行任务（写代码 / 调试 / 文档）       │
                    │         ↓                                │
                    │  4. 自我审查 + 请求 Agent 评审            │
                    │         ↓                                │
                    │  5. 开 PR → 自动合并（如通过）            │
                    │         ↓                                │
  人类只在需要  ◀──  │  6. 清理计划文件（保持仓库整洁）          │
  判断时介入         │         ↓                                │
                    │  → 循环（下一个任务）                     │
                    └──────────────────────────────────────────┘
```

<v-clicks>

- **新鲜上下文**：每次任务新开会话，避免历史对话污染推理
- **反压控制**：CI 失败、测试不通过 → Agent 自动修复，不阻塞其他任务
- **临时计划**：任务计划提交到仓库，完成后归档，下次 Agent 可读取历史

</v-clicks>

<!-- 演讲者备注：Ralph Loop 的核心是"廉价的纠错" + "充分的可观测性" -->

# Planner → Generator → Evaluator 架构

**Anthropic Labs 提炼的三层 Agent 协作模式**

<v-click>

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Planner Agent           Generator Agent    Evaluator Agent │
│  ──────────────          ───────────────    ──────────────  │
│  接收需求                 读取计划             检查输出        │
│  拆解任务                 查阅 AGENTS.md      运行测试        │
│  写执行计划 ──────────▶   生成代码     ──────▶ 发现偏差        │
│  存入仓库                 开 PR                请 Planner    │
│                                               修订计划       │
│                                                   │         │
│                                                   └─────────│
└─────────────────────────────────────────────────────────────┘
```

</v-click>

<v-clicks>

- **Planner**：理解意图，生成可执行计划，写入 `docs/exec-plans/active/`
- **Generator**：读取计划，查阅地图，生成代码 PR
- **Evaluator**：独立评审，发现偏差，推动计划迭代

</v-clicks>

<v-click>

> 三个 Agent 相互制衡，人类只审核 Evaluator 无法决策的边界情况。

</v-click>

<!-- 演讲者备注：这个架构已经在 Anthropic 内部多个项目中验证。关键是 Evaluator 的独立性 -->

# Harness 反模式

<style>
li { font-size: 0.83rem !important; }
code { font-size: 0.8rem !important; }
</style>

<v-clicks>

**🚫 反模式 1：把 AGENTS.md 当百科全书**
```
❌ 1000行 AGENTS.md，每个细节都记录在里面
✅ ~100行地图，指向 docs/ 下的具体文档
```
→ 过大的指令文件会挤掉任务代码，让 Agent 开始模式匹配而非有意导航

**🚫 反模式 2：规范只靠文档约束**
```
❌ 在 README 里写"禁止直接 fetch，请用 src/api/"
✅ 自定义 ESLint 规则，CI 直接阻断违规代码合并
```
→ 文档会被遗忘和过期；lint 规则是机械强制执行的

**🚫 反模式 3：知识存在仓库外**
```
❌ 架构决策在 Slack 里讨论，没有落地为文档
✅ 讨论结果转化为 docs/design-docs/xxx.md 并提交
```
→ Agent 看不到的等于不存在，Slack 对 Agent 是黑洞

**🚫 反模式 4：让技术债自然累积**
```
❌ 等"有空的时候"再清理 AI 产生的坏模式
✅ 定期运行 refactor Agent，自动发起清理 PR
```
→ 坏模式会被 Agent 复制放大，债务呈指数增长

</v-clicks>

# 三层工程：协同全景

```
┌────────────────────────────────────────────────────────────────────┐
│                      完整的 AI 工程栈                               │
│                                                                    │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  Layer 3: Harness 工程                                        │  │
│  │  设计环境 · 机械化约束 · Agent 可读性 · Ralph Loop            │  │
│  ├──────────────────────────────────────────────────────────────┤  │
│  │  Layer 2: Context 工程                                        │  │
│  │  ICIO 框架 · CLAUDE.md · 精准投喂 · 首尾定律                  │  │
│  ├──────────────────────────────────────────────────────────────┤  │
│  │  Layer 1: Prompt 工程                                         │  │
│  │  Zero-Shot → CoT → Extended Thinking · Structured Output     │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                    │
│  Prompt 决定任务方向   Context 决定输出质量   Harness 决定持续能力   │
└────────────────────────────────────────────────────────────────────┘
```

<v-clicks>

- **只有 Prompt**：输出随机，不可复用，团队无法共享
- **Prompt + Context**：输出稳定，单次任务质量高
- **Prompt + Context + Harness**：<span v-mark.underline.yellow="3">**持续交付，规模化，自我演化**</span>

</v-clicks>

<!-- 演讲者备注：这三层不是替代关系，而是叠加关系。每一层都是必要的，但不充分 -->
