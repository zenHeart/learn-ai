# 场景一：裸 Prompt vs 装备 Prompt

<v-click>

**第一步：没有任何 Context 时**

```
用户：帮我写一个 Vue3 表格组件，支持分页和排序

AI 输出：
<template>
  <div>
    <table>
      <tr v-for="item in data" :key="item.id">
        <td>{{ item.name }}</td>
      </tr>
    </table>
  </div>
</template>

<script>
export default {                   // ❌ Options API
  data() {                         // ❌ 违反团队规范
    return { items: [] }
  }
}
</script>
```

</v-click>

# 场景一：装备 Prompt 的输出

**第二步：使用 ICIO + 全局约束**

```markdown
@src/components/BaseTable.vue     ← @引用现有组件作为 Few-Shot
@.cursorrules  或 @.cursor/rules/   ← 注入项目规范

## Instruction
请参照 BaseTable.vue 的实现风格，生成 DataGrid 组件。

## Context
- 组件用于展示用户列表，数据由父组件通过 props 传入
- 需要支持：多列排序（asc/desc 切换）、前端分页（每页 20 条）
- 排序状态通过 emit 通知父组件，便于 URL 同步

## Input（数据结构）
interface User {
  id: number; name: string; email: string;
  role: 'admin' | 'user'; createdAt: string;
}

## Output
输出完整 .vue 文件，包含 TypeScript 类型定义和 Vitest 单元测试骨架
```

<v-click>

**AI 输出（符合规范）**：

```vue
<script setup lang="ts">              // ✅ Composition API
import { ref, computed } from 'vue'
import type { User } from '@/types'

interface Props {
  data: User[]                        // ✅ 明确类型
  loading?: boolean
}

const props = defineProps<Props>()
const emit = defineEmits<{           // ✅ 明确 emit 类型
  sortChange: [field: keyof User, order: 'asc' | 'desc']
}>()

const sortField = ref<keyof User>('createdAt')
const sortedData = computed(() => [...props.data].sort(...))
</script>
```

</v-click>

# 场景一：关键实践

<v-clicks>

**1. @file 引用替代文字描述**
- 直接引用 `@BaseTable.vue`，而非"参考我们的表格风格"
- 真实代码 >> 语言描述

**2. 项目规范自动注入（.cursorrules 或 .cursor/rules/）**
- 无需每次手动告诉 AI"用 Composition API"
- 团队所有人问出来的结果风格一致

**3. 明确数据结构（TypeScript Interface）**
- 在 Input 部分给出 Interface，AI 不会自造字段
- 减少幻觉字段的出现概率

**4. Output 要求测试骨架**
- 强制 TDD 意识，AI 同时生成测试文件骨架

</v-clicks>

# 场景二：裸 Context vs 精准 Context

**背景**：nn-client-all 凌晨偶发 crash，堆栈指向 WebSocket 连接管理层

# 场景二：精准 Context 切取策略

**正确做法：外科手术式精准切取**

<v-clicks>

**Step 1：定位核心调用链路**
- 看堆栈，找最近的业务代码帧（不是 node_modules）
- 通常只有 3-5 个关键函数

**Step 2：切取最小代码集合**
- 只复制报错的函数 + 直接调用它的函数
- 不要整个文件，只要相关的方法

**Step 3：结构化错误信息**
- 错误类型 + 完整堆栈
- 触发条件（什么情况下出现）
- 已尝试过的方案

</v-clicks>

# 场景二：精准 Context 示例

```xml
<error>
类型：UnhandledPromiseRejection
错误信息：Cannot read property 'send' of null
触发条件：网络切换时（WiFi → 4G），概率约 1/50

堆栈（只保留业务层）：
  WebSocketManager.reconnect (ws-manager.ts:142)
  ConnectionPool.handleDisconnect (pool.ts:87)
  EventEmitter.emit (event.ts:23)
</error>

<code>
// ws-manager.ts:135-155
async reconnect(clientId: string) {
  const ws = this.connections.get(clientId)  // 可能是 null？
  await new Promise(r => setTimeout(r, 1000))
  ws.send(JSON.stringify({ type: 'ping' }))  // ← 报错行
}

// pool.ts:82-95
handleDisconnect(clientId: string) {
  this.pool.delete(clientId)               // ← 先删了
  this.manager.reconnect(clientId)         // ← 再重连，此时已经 null
}
</code>

<question>
分析竞态条件根因，给出类型安全且不引入 breaking change 的修复方案
</question>
```

# 场景二：AI 精准输出

<v-click>

**AI 分析（直指根因）**：

```
根因分析：
这是典型的 TOCTOU（Time-of-Check-Time-of-Use，检查时间与使用时间之间存在时间差，窗口期内状态可能已改变）竞态条件。

1. handleDisconnect 先从 pool 删除 connection（line 84）
2. 之后调用 reconnect，从 connections.get() 获取
3. 但 pool 和 connections 是两个独立数据结构
4. 重连期间再次触发 disconnect，ws 已为 null

修复方案：

async reconnect(clientId: string) {
  const ws = this.connections.get(clientId)
  if (!ws) {                               // ✅ null 检查
    console.warn(`[WS] No connection for ${clientId}`)
    return
  }
  await new Promise(r => setTimeout(r, 1000))
  if (this.connections.has(clientId)) {    // ✅ 二次确认
    ws.send(JSON.stringify({ type: 'ping' }))
  }
}
```

</v-click>

# 场景二：关键实践

<v-clicks>

**1. 堆栈 → 最小代码路径**
- 只提取 crash 相关的 3-5 个函数，保留行号

**2. 错误信息结构化**
- 错误类型、触发条件、已尝试方案 → 分块清晰

**3. 明确问题边界**
- "类型安全且不引入 breaking change" → 约束 AI 解决方向

**4. 不要问"有什么问题"**
- 要问"分析这个竞态条件的根因" → 精确定位

</v-clicks>

# 两个场景的共同结论

<v-clicks>

| 因素 | 低质量 Context | 高质量 Context |
|------|--------------|---------------|
| 信息量 | 越多越好（堆文件） | 最小有效集（精准切取） |
| 结构 | 自然语言描述 | XML/Markdown 结构化 |
| 规范 | 每次手动说明 | `.cursorrules` 或 `.cursor/rules/` 自动注入 |
| 示例 | 无示例 | @引用现有代码 |
| 问题 | 开放式 | 聚焦式（"分析根因"） |

</v-clicks>

<v-click>

> **核心结论**：Context Engineering 不是让你写更多，而是让你写得更准。
> 精准 > 全面，结构 > 堆砌，约束 > 自由。

</v-click>
