# 03 三层协同

化学反应，不是简单相加

---

# 三层各自的盲区

<v-clicks>

| 只有好 Prompt | 只有好 Context | 只有好 Harness |
|--------------|----------------|----------------|
| 模型自由发挥，每次结果随机 | 塞太多，淹没关键信息，Token 成本失控 | 工具链很强，但问不对问题 |
| 无法团队对齐，Prompt 散落各处 | 过期信息不被清理，误导 AI | Agent 读不到规范，执行结果不一致 |

</v-clicks>

<v-click>

> **结论**：三层缺一不可，且必须协同设计。

</v-click>

---

# 全景图：三层如何协同

<v-click>

```
第一步：Prompt（任务说明书）
└─ 清晰指令：角色 + 约束 + 输出格式

第二步：Context（项目笔记本）
└─ 精准注入：只给 AI 当前任务需要的 8K Token

第三步：Harness（工作环境）
└─ 工具链：读文件 / 写代码 / 跑测试 / 开 PR
   + 记忆：Session / Project / User
   + 循环：执行 → 观测 → 修正 → 验证
```

</v-click>

<v-click>

> **Cursor 为什么强**：不是模型更好，而是三层同时到位的产品化。

</v-click>

# 真实案例：Cursor 修 Bug 的 30 秒

```
场景：用户报告"登录失败"
Cursor 实际执行链路：

Prompt 层：
  "修一下登录失败" → 拆解为"定位失败点 → 修复 → 验证"

Context 层：
  自动注入相关文件（login.vue、auth.ts）
  自动注入 git diff（最近谁改了什么）
  自动注入 CI 日志（哪一步失败）

Harness 层：
  Tool: read_file → search → edit_file → run_tests → create_pr
  Memory: 项目规范（TypeScript strict、API 统一走 src/api/）
  Loop: 修代码 → 跑测试 → 发现问题 → 继续修 → CI 通过 → 自动合并
```

<v-click>

> **30 秒内**：定位 → 修复 → 验证 → 合并。全程无需人工介入。

</v-click>
