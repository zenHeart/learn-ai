### **Q1: 为什么非要套一层 MCP 增加复杂度？我直接在脚本里调 API 不是更简单吗？**

A: **对于"一次性脚本"，直接调 API 确实更简单；但对于"生态级集成"，MCP 是降维打击。**

<div class="mt-6 space-y-5 text-sm">

- **复利效应**: [Anthropic 在官方博客指出](https://www.anthropic.com/news/model-context-protocol)，只要你写了一次 Jira 的 MCP Server，明天不管是 Cursor、Windsurf、Claude 甚至后天的某个新 IDE 出现，**不需要改一行代码**，所有的 AI 工具都能瞬间具备操作 Jira 的能力。
- **权限收口**: 如果直接让 AI 写 Python 脚本去调数据库，意味着你要把生产环境的账号密码直接暴露给大模型（极度危险）。而 MCP 架构把密钥和具体执行逻辑封装在本地 Server 里，对大模型完全隔离。

</div>

---

### **Q2: 为什么我写的 Skill，AI 经常乱用或者根本不听我的规则？**

A: **因为你把它当成了人，而不是当成了编译器。**

<div class="mt-6 space-y-5 text-sm">

- **官方铁律**: 根据 [Anthropic 的 Tool Use 最佳实践](https://docs.anthropic.com/en/docs/build-with-claude/tool-use)，不要对 AI 说"请尽量避免使用..."，而是要说"**绝对禁止**在 XX 情况下使用，否则会导致严重错误"。
- **Schema 约束**: 把所有能用 `enum` (枚举) 框死的地方全部框死。如果你让 AI 传一个自由的字符串，它就会产生幻觉。给它明确的下拉菜单，它就不会出错。

</div>

### **Q3: 自己开发 MCP Server 门槛高吗？**

A: **非常低，几乎等同于写一个普通的 Node.js / Python 函数。**

<div class="mt-6 space-y-4 text-sm">

1. **SDK 完善**: 官方提供了 `@modelcontextprotocol/sdk` (TS) 和 `mcp` (Python)
2. **特别是 Python 的 FastMCP**: 它的心智模型完全借鉴了 FastAPI。你只需要写一个普通的 Python 函数，加一个 `@mcp.tool()` 装饰器，它就自动变成了一个标准的 MCP Tool

</div>

::right::

### **Q4: `.cursorrules` (规则) 和 CLI 里的 Skill 到底有什么区别？**

A: **触发机制和作用范围不同。**

<div class="mt-6 space-y-4 text-sm">

1. **`.cursorrules` (被动守则)**: 只要在这个项目里写代码，它就会作为 Background Context 一直存在，默默约束 AI 的所有行为
2. **Skill (主动大招)**: 是一套包含具体步骤的"工作流"。它平时潜水，只有当你敲下 `/audit` 或者 `/deploy` 时才被唤醒，去执行一套复杂的 SOP

</div>

---

### **Q5: MCP 只能用于本地（stdio）吗？可以跨网络调用吗？**

A: **完全可以跨网络，并且 2025-03-26 起官方推荐用 Streamable HTTP（旧 HTTP+SSE 已废弃）。**

<div class="mt-6 space-y-4 text-sm">

1. **企业内网部署**: 把 MCP Server 部署在公司内网（如 10.0.0.1），通过 **Streamable HTTP** 暴露 —— HTTP POST + 可选 SSE 流，OAuth/Bearer/API Key 鉴权，`Mcp-Session-Id` 头维持会话。
2. **安全红线**: Streamable HTTP Server **MUST** 校验 `Origin` 头防 DNS rebinding；本地仅绑 `127.0.0.1`；推荐 OAuth 取 token，避免长期密钥散落在客户端。
3. **集中管控**: 全体研发的 Cursor / Claude / VS Code 都可以连到这个统一节点，便于版本管理和服务端权限审计。

</div>

::right::

### **Q6: 遇到复杂的流水线，我是写一个大 Skill，还是写多个小 Skill？**

A: **坚决写多个小 Skill。**

<div class="mt-6 space-y-4 text-sm">

1. **控制变量**: 像写微服务一样写 Skill。一个叫 `run-linter`，一个叫 `build-image`
2. **编排机制**: 让一个高等级的 Agent 去根据当前状态，按需调用这些原子化的 Skill。把它们揉成一团只会导致 AI 经常在中间某一步因为超时而全盘崩溃

</div>

---

### **Q7: MCP + Skill 组合使用的最佳实践是什么？**

A: **MCP 决定能力上限，Skill 决定稳定下限。**

<div class="mt-6 space-y-4 text-sm">

1. **MCP 负责对外**: 封装对数据库、API、文件系统的访问能力（Tools / Resources / Prompts 三大原语）
2. **Skill 负责对内**: 定义团队规范、SOP 流程、错误处理策略，靠 Progressive Disclosure 不爆 Context
3. **决策口诀**: 需要伸手到外部 → MCP；需要按规范走流程 → Skill；都要 → 组合

</div>

---

### **Q8: MCP 远程 Server 怎么做鉴权和会话管理？** <span class="text-xs text-slate-400 font-normal">🏢 企业级</span>

A: **Streamable HTTP + OAuth + Origin 校验三件套。**

<div class="mt-4 space-y-3 text-sm">

1. **鉴权**: HTTP `Authorization: Bearer <token>` 头，token 通过 OAuth flow 获取（避免长期密钥散落客户端）。生产环境推荐对接公司 SSO / OIDC。
2. **会话**: 服务端在 `initialize` 响应里下发 `Mcp-Session-Id`（UUID 或 JWT），客户端后续请求都带上；服务端可主动 `HTTP 404` 让客户端重连。
3. **DNS rebinding 防御**: Server **MUST** 校验 `Origin` 头白名单；本地仅绑 `127.0.0.1` 不绑 `0.0.0.0`。
4. **Audit**: 记录每次 `tools/call` 的 caller、arguments、result，便于安全审计。

</div>

---

### **Q9: Skill 在团队里怎么版本化和灰度上线？** <span class="text-xs text-slate-400 font-normal">🏢 企业级</span>

A: **Git 管理 + `metadata.version` + 分阶段下发。**

<div class="mt-4 space-y-3 text-sm">

1. **版本号**: 在 `SKILL.md` frontmatter 里写 `metadata.version: "2.1.0"`，遵循 SemVer。
2. **存放**: 团队级 Skill 放专用 Git 仓库（如 `company-skills`），通过内部 npm / pip 包或 git submodule 分发。
3. **灰度**: 先 5% 工程师手动安装试用，CI 收集反馈 → 扩到 30% → 全量。Slack 频道收集吐槽。
4. **回滚**: Skill 写错了 description 导致疯狂误激活？Git revert + `claude skill remove` 即可，30 秒回滚。
5. **变更通知**: 每次 SKILL.md 更新自动生成 changelog，推送到 Slack `#skills-changelog`。

</div>

### **Q10: 什么时候应该写 Skill，什么时候写 Sub-agent？** <span class="text-xs text-slate-400 font-normal">🏢 企业级</span>

A: **Skill = 知识 + 流程；Sub-agent = 独立上下文 + 独立工具集。边界在"是否需要隔离"。**

<div class="mt-4 grid grid-cols-2 gap-3 text-xs">

<div class="theme-skill">

### Skill 适合

* 有明确的 SOP 流程
* 需要被自动激活（description 匹配）
* 共享主对话上下文
* 工序在 5–30 分钟内完成
* 例：code-review / release-check / weekly-report

</div>

<div class="theme-mcp">

### Sub-agent 适合

* 需要独立上下文避免污染主对话
* 用专属工具集（如限制只能 Read 不能 Write）
* 任务可并行（多个 Sub-agent 同时跑）
* 例：parallel-pr-reviewer / data-explorer / sandbox-tester

</div>

</div>

<div class="mt-3 text-xs text-slate-500">
经验法则：先用 Skill；当主对话上下文被任务"污染"得太严重，或需要并行执行多个独立子任务时，再升级为 Sub-agent。
</div>
