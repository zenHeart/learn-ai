# Demo A · TypeScript 版 (2/4) <span class="text-xs text-slate-400 font-normal">📎 现场二选一</span>

```ts
// server.ts
import 'dotenv/config'
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'
import { z } from 'zod'

const server = new McpServer({ name: 'jira-mcp', version: '1.0.0' })
const JIRA_BASE = process.env.JIRA_BASE!
const JIRA_TOKEN = process.env.JIRA_TOKEN!

async function jira(path: string) {
  const r = await fetch(`${JIRA_BASE}${path}`, {
    headers: { Authorization: `Bearer ${JIRA_TOKEN}` },
  })
  if (!r.ok) throw new Error(`Jira ${r.status}: ${await r.text()}`)
  return r.json()
}

server.tool(
  'get_jira_issue',
  '按工单 ID 查询 Jira 工单详情，返回 status / assignee / description / labels。当用户提到 PROJ-1234 这类工单号或要求"查工单状态"时调用。',
  { issueId: z.string().regex(/^[A-Z]+-\d+$/).describe('如 PROJ-1024') },
  async ({ issueId }) => {
    try {
      const data = await jira(`/rest/api/3/issue/${issueId}`)
      return { content: [{ type: 'text', text: JSON.stringify({
        status: data.fields.status.name,
        assignee: data.fields.assignee?.displayName ?? '未分配',
        summary: data.fields.summary,
      }, null, 2) }] }
    } catch (e: any) {
      return { content: [{ type: 'text', text: `查询失败: ${e.message}` }], isError: true }
    }
  },
)

server.resource('jira-schema', 'jira://schema', async () => ({
  contents: [{ uri: 'jira://schema', mimeType: 'application/json',
              text: JSON.stringify(JIRA_FIELD_SCHEMA, null, 2) }],
}))

await server.connect(new StdioServerTransport())
```

<!-- 演讲者备注：
现场二选一：前端团队主讲 TS 版（这屏），后端团队主讲 Python 版（下一屏）。完整代码已放 examples/jira-mcp-server/。
关键讲解点：
① zod schema 的 regex 校验 —— 在 LLM 调用前就拒绝错误格式
② try/catch + isError: true —— Tool 不要抛异常，而是把错误装在 result 里返给 AI
③ AbortSignal.timeout(10_000) —— 防止 Tool 卡死整个对话
-->

# Demo A · Python (FastMCP) 版 (3/4) <span class="text-xs text-slate-400 font-normal">📎 现场二选一</span>

```python
# server.py
import os, json, requests
from mcp.server.fastmcp import FastMCP
from dotenv import load_dotenv

load_dotenv()
mcp = FastMCP("jira-mcp")
JIRA_BASE = os.environ["JIRA_BASE"]
JIRA_TOKEN = os.environ["JIRA_TOKEN"]

def _jira(path: str) -> dict:
    r = requests.get(f"{JIRA_BASE}{path}",
                     headers={"Authorization": f"Bearer {JIRA_TOKEN}"},
                     timeout=10)
    r.raise_for_status()
    return r.json()

@mcp.tool()
def get_jira_issue(issue_id: str) -> dict:
    """按工单 ID 查询 Jira 工单详情。

    Args:
        issue_id: 形如 PROJ-1024 的工单号
    Returns:
        含 status / assignee / summary / description 的字典
    """
    data = _jira(f"/rest/api/3/issue/{issue_id}")
    return {
        "status": data["fields"]["status"]["name"],
        "assignee": (data["fields"].get("assignee") or {}).get("displayName", "未分配"),
        "summary": data["fields"]["summary"],
    }

@mcp.resource("jira://schema")
def jira_schema() -> str:
    """返回 Jira 工单字段 schema（JSON）。"""
    return json.dumps(JIRA_FIELD_SCHEMA, indent=2)

if __name__ == "__main__":
    mcp.run()
```

<div class="mt-2 text-xs text-slate-500">
FastMCP 自动从 docstring + 类型注解生成 description 和 inputSchema。
</div>

<!-- 演讲者备注：
对比 TS 版的关键差异：
① docstring 自动变成 Tool description —— DX 极佳
② Literal 类型自动变成 enum 约束
③ 抛异常 FastMCP 会自动包装成 isError: true，无需手写 try/catch
强调："如果你写过 FastAPI，FastMCP 几乎零学习成本。"
-->

# Demo B · Code Review Skill (1/2)

### 需求

团队 Code Review 老是格式不统一、漏看 SQL 注入和 N+1。把 SOP 沉淀成 Skill，自动激活。

```
~/.claude/skills/code-review/
├── SKILL.md
└── references/
    ├── SQL_INJECTION.md
    └── N_PLUS_ONE.md
```

```markdown
<!-- SKILL.md -->
---
name: code-review
description: |
  审查 PR diff，重点检查 SQL 注入、N+1 查询、敏感日志泄露与异常处理覆盖率。
  当用户要求"代码审查 / review PR / 检查这次改动 / code review"时调用。
allowed-tools: Bash(git:diff) Bash(git:log) Read
license: MIT
metadata:
  team: backend-platform
  version: "1.0.0"
---

## When to use
当用户传入 PR 编号、贴出 diff，或要求"审查这次改动"时。

## How to do
1. 通过 `git diff` 拉到改动（或调用 GitHub MCP `get_pull_request`）
2. 按以下顺序检查：
   - **SQL 注入**：参考 `references/SQL_INJECTION.md`，重点看字符串拼接 + ORM 原生查询
   - **N+1**：参考 `references/N_PLUS_ONE.md`，循环里有 ORM 调用即标红
   - **敏感日志**：搜 `password|token|secret|cookie` 在日志输出附近
   - **异常处理**：每个 try 是否吃掉了关键错误
3. 输出结构化报告：风险等级 / 行号 / 建议修改

## Common edge cases
- diff 超过 500 行：先按文件分批，避免一次性塞爆 Context
- 二进制文件：跳过
- 测试代码：放宽 SQL 注入要求（一般是 mock）
```

# Demo C · MCP × Skill 联动发版 (1/2)

### 需求

每次发 staging 都要：跑测试 → 检查覆盖率 → 改 changelog → 打 tag → 通知 Slack。

```markdown
<!-- ~/.claude/skills/release-staging/SKILL.md -->
---
name: release-staging
description: |
  执行 staging 环境发版的完整流程。
  当用户输入 /release-staging 或说"发个 staging / 部署预发"时调用。
allowed-tools: Bash(npm:*) Bash(git:*) Read Edit Write
metadata:
  version: "2.1.0"
---

## 执行步骤

1. **跑测试** `npm test` —— 失败立即停止并报告
2. **检查覆盖率** `npm run test:coverage`
   - 如果 < 80%，**stop** 并问用户是否强制继续
3. **改 changelog**
   - 调用 git MCP `git_log` 拉最近 commit
   - 按 conventional commit 分类（feat/fix/refactor）
   - 写入 `CHANGELOG.md` 顶部
4. **打 tag** `git tag v$(VERSION)`
5. **推送** `git push origin v$(VERSION)`
6. **通知 Slack**
   - 调用 slack MCP `send_message`
   - 频道 #release，含 changelog 链接和 commit 列表

## Common edge cases
- 工作区有未提交改动 → stop
- 测试超时（>5min）→ 标记并询问
- Slack 通知失败 → 不阻断，记录到本地日志
```

# 踩坑警示 · 真实案例集

<div class="grid grid-cols-2 gap-3 mt-3 text-sm">

<div class="col-bad">

### ⚠️ JSON-RPC timeout 卡死

**现象**：MCP Server 调内部 API 超时，整个对话卡 60s+<br>
**根因**：默认无超时限制<br>
**修复**：fetch 显式 `signal: AbortSignal.timeout(10_000)`，Tool 返回 `isError: true` 而非抛出

</div>

<div class="col-bad">

### ⚠️ Skill 权限过大

**现象**：`allowed-tools: Bash(*:*) Write` 让 Skill 能跑任意命令<br>
**根因**：图省事写 wildcard<br>
**修复**：精确到 `Bash(git:tag) Bash(npm:test) Read`

</div>

<div class="col-bad">

### ⚠️ description 关键词撞车

**现象**：两个 Skill 都含 "测试" 关键词，互相抢激活<br>
**根因**：description 不够独特<br>
**修复**：加上限定词："**单元**测试" vs "**端到端**测试"

</div>

<div class="col-bad">

### ⚠️ Server 没校验 Origin

**现象**：远程 MCP Server 上线后被恶意网页通过 DNS rebinding 调用<br>
**根因**：Streamable HTTP 没校验 `Origin` 头<br>
**修复**：白名单 `Origin`，本地仅绑 `127.0.0.1`

</div>

</div>

<div class="mt-4 text-center text-sm text-slate-500">
所有踩坑都来自团队真实经历 —— 一定要写到 SKILL.md 的 "Common edge cases" 章节。
</div>
