# 实战场景 1：新项目冷启动 (Intent to Architecture)

> 从 0 → 1 构建项目：输入是**模糊意图**（脑暴想法 / 竞品参考），输出是**可运行的项目**。<VibeExample id="4.1" />


<div class="pair-grid">

::: div{.pair-card.blue}
🗺️ Plan（规划）{.card-title}

输入：意图描述{.card-meta}

用 **Plan Mode** 驱动 AI 生成 `user-guide.md`（用户手册）， 明确产品功能和用户的使用方式。{.card-body}

产出：user-guide.md{.card-out}
:::

::: div{.pair-card.purple}
🔍 Assess（对齐）{.card-title}

评估：user-guide.md{.card-meta}

检查使用方式是否解决了用户痛点， 进一步生成架构设计和待办事项。{.card-body}

产出：design.md + todo.md{.card-out}
:::

::: div{.pair-card.green}
⚡ Implement（执行）{.card-title}

粒度：每次 = 一个用户故事{.card-meta}

按 Todo 逐个用户故事执行： 按照设计架构，可以多 Agents 并行执行，先完成 MVP, 在迭代优化。{.card-body}

产出：可运行的 MVP{.card-out}
:::

::: div{.pair-card.amber}
✅ Review（验收）{.card-title}

卡点：每模块完成后{.card-meta}

通过测试，人工验证，对照 todo.md 检查是否完成；架构偏差 → 回溯 Plan。{.card-body}

最终：可运行项目{.card-out}
:::

</div>

# 实战场景 2：feature 需求开发 (PRD to PR)

> 已有项目新增功能：输入是 **PRD + 设计稿**，输出是经过测试验证的**功能 PR**。<VibeExample id="4.2" />

<div class="pair-grid">

::: div{.pair-card.blue}
🗺️ Plan（规划）{.card-title}

输入：PRD.md + UI 设计稿{.card-meta}

@ 引用 PRD.md 和 UI 截图或者 figma MCP，用 **Plan Mode** 生成 `design.md`（接口/数据流/组件边界）和 `todo.md`（Story 拆解）。{.card-body}

产出：design.md + todo.md{.card-out}
:::

::: div{.pair-card.purple}
🔍 Assess（对齐）{.card-title}

评估：design.md{.card-meta}

Review 接口字段完整性、数据流向、组件归属；确认 todo 粒度可执行。方案不合理 → 返回 Plan 补充。{.card-body}

产出：高质量 design.md{.card-out}
:::

::: div{.pair-card.green}
⚡ Implement（执行）{.card-title}

粒度：每次 = 一个功能块{.card-meta}

按 `todo.md` 逐条执行，@ 引用 `AGENTS.md` 约束产出规范。复杂 Story 可嵌套子 PAIR 循环。{.card-body}

产出：增量功能+测试用例{.card-out}
:::

::: div{.pair-card.amber}
✅ Review（验收）{.card-title}

卡点：每个功能块完成后{.card-meta}

功能测试通过；对比 Diff 关注边缘情况；逻辑缺陷 → 回溯 Plan 修正 design.md。{.card-body}

最终：功能 PR{.card-out}
:::

</div>

# 实战场景 3：hotfix Bug 修复 (Crash to PR)

> 从线上问题到闭环修复：输入是 **Crash 堆栈 / 日志**，输出是带 Repro Test 的**修复 PR**。 <VibeExample id="4.3" />

<div class="pair-grid">

::: div{.pair-card.blue}
🗺️ Plan（归因）{.card-title}

输入：Crash 堆栈 / Sentry / 业务日志 / bug 描述{.card-meta}

注入堆栈 + 日志，让 AI 跨文件追踪调用链，生成 `root-cause.md`（根因分析）和 `fix-todo.md`（修复计划）。{.card-body}

产出：fix.md{.card-out}
:::

::: div{.pair-card.purple}
🔍 Assess（评估）{.card-title}

评估：fix.md{.card-meta}

确认根因是否准确；修复面是否最小化（避免牵连无关模块）；归因存疑 → 返回 Plan ，补充更多信息。{.card-body}

产出：fix.md + todo.md{.card-out}
:::

::: div{.pair-card.green}
⚡ Implement（修复）{.card-title}

粒度：每次 = 一个用例{.card-meta}

先写复现测试（固化 Bug），再修复代码，确保测试由红变绿。按 `todo.md` 原子化执行。{.card-body}

产出：修复 + test{.card-out}
:::

::: div{.pair-card.amber}
✅ Review（验收）{.card-title}

卡点：每处修复完成后{.card-meta}

Repro Test 通过；无新 Warning；全量测试无 regression；PR 描述含复现步骤与根因说明。{.card-body}

最终：修复 PR{.card-out}
:::

</div>

<div class="summary-grid">

::: div{.summary-card}
📚 PAIR 工作流 SOP

1. **Plan (计划)** 明确需求，制定详细计划
2. **Assess (评估)** 进入 Plan Mode 确认 AI 理解
3. **Implementation (实施)** 标准化执行，Agent Loop
4. **Review (评审)** 人工评审兜底，边界/性能/安全审查
:::

::: div{.summary-card}
🔄 双微循环机制

- **理解对齐循环** (Plan ↔ Assess): 迭代直到 AI 完全理解
- **Checkpoint 验证循环** (Implementation ↔ Review): 每个小步骤后的验证
- **回退路径**: 理解偏差 → Plan / 需精细化 → Plan
:::

::: div{.summary-card}
🧠 Vibe 核心：从 Builder 到 Reviewer

1. **意图第一** 代码只是意图的副产品，关注"你想要什么"。
2. **审查为先** 你的价值不在于打字快慢，而在于识别"正确的代码"。
3. **拒绝手动** 只要能通过规则/测试约束 AI 的，绝不自己动手改代码。
:::

::: div{.summary-card}
📊 组织与持续改进

- 通过 `AGENTS.md` 管理项目和通用规范
- 利用 commands/rules 沉淀团队 prompt
- 利用 SKILL 沉淀各种工作流
- 利用 MCP 提供大模型内网信息
:::

</div>
