# 5. 实战场景

## 场景 1：代码生成与重构 (遵循 ICIO)

**痛点**: 每次生成的组件不符合公司规范，需要反复修改。
**解法**: 通过 ICIO 框架，一次性把“前端技术栈”和“规范”交代清楚。

```markdown
【Instruction】
作为前端架构师，重构以下过时的 React Class 组件为 Functional Component。

【Context】
- 技术栈：React 18, TypeScript, TailwindCSS v4
- 规范：必须使用 hooks 管理状态，不可使用 PropTypes，必须导出类型。

【Input Data】
\`\`\`jsx
class UserCard extends React.Component { ... }
\`\`\`

【Output】
- 只输出代码，包含必要的注释。
```

## 场景 2：使用 CoT 攻克复杂逻辑 (正则/SQL)

**痛点**: 大模型直接写复杂的正则表达式或长 SQL 经常翻车（如漏掉边界条件）。
**解法**: 强制模型“一步步思考”(Chain-of-Thought)，拆解匹配规则。

```markdown
请帮我写一个正则表达式，匹配标准的中国大陆手机号码。

要求：
为了保证准确性，请你按照以下步骤思考（Let's think step by step）：
1. 分析号段规则（以 1 开头）
2. 分析第二位数字的合法范围（3-9）
3. 分析总长度限制（11 位）
4. 分析是否需要处理首尾空白符

最后，请将正则结果包裹在 <regex> 标签中，将测试用例包裹在 <test> 标签中。
```

<div class="mt-4 p-4 bg-green-50 text-green-800 rounded-lg text-sm">
<strong>效果</strong>: 模型会先输出推理过程，自我纠错，最终给出的正则和 SQL 准确率飙升。
</div>

## 场景 3：利用 RAG 落地私有库问答机器人

**痛点**: 模型不知道公司内部的接口文档和业务逻辑。
**解法**: 不用微调模型，使用 LlamaIndex 搭建本地 RAG 系统。

```python
# 核心落地代码框架 (只需 5 行)
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

# 1. 加载内部私有文档 (Markdown, PDF 等)
docs = SimpleDirectoryReader("./company-api-docs").load_data()

# 2. 自动分块、调用 Embedding 模型，并建立内存向量库
index = VectorStoreIndex.from_documents(docs)

# 3. 接收用户提问，自动进行向量检索，并带入 Context 给大模型
engine = index.as_query_engine()
answer = engine.query("调用创建订单接口时，签名算法是怎么算的？")

print(answer)
```
