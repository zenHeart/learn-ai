# 迁移到 AI 优先架构

**问题**: 你有一个单体应用。管理层想要“处处有 AI”。
**风险**: AI 是非确定性的。它可能会破坏核心工作流。

## 模式：AI Sidecar (边车)

不要替换逻辑。**增强**它。

### 阶段 1: “助手” (低风险)
添加一个悬浮操作按钮 (FAB)，打开聊天机器人。
- **访问权限**: 对文档的只读访问权限。
- **影响**: 如果它失败了，应用仍然可以工作。

### 阶段 2: “副驾驶” (中等风险)
在表单字段内添加 AI。
- **功能**: “自动填充此表单”按钮。
- **影响**: 用户必须审查并点击“保存”。人仍在循环中。

### 阶段 3: “智能体” (高风险)
AI 在后台采取行动。
- **功能**: “自动批准 < $50 的退款请求”。
- **安全性**: 要求“置信度分数” > 99%。如果是 98%，则路由给人工。

## 风险缓解

| 风险 | 缓解措施 |
| :--- | :--- |
| **幻觉** | 使用 RAG + 引用。永远不要让 AI 猜测。 |
| **成本爆炸** | 对每个用户实施严格的速率限制。 |
| **数据泄露** | PII 编校中间件。 |
| **UX 困惑** | 对 AI 内容使用不同的 UI 风格（例如，紫色火花）。 |
